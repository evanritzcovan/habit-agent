🧠 Product Overview

A mobile habit app that helps users build and break habits using structured AI-generated plans, with tracking, streaks, and lightweight coaching.

The app focuses on:
    - Reducing friction to take action
    - Providing structured, actionable plans (not generic advice)
    - Reinforcing consistency via streaks and daily views

🎯 Core Principles
    - Clarity over complexity (simple UX, structured plans)
    - Consistency over intensity (daily execution focus)
    - AI as assistant, not replacement
    - Monetization from day one

📁 Repository layout (single project, one package.json)
    - One repo root: Expo app code (app/, components/, lib/, etc.), Supabase (supabase/), shared Zod (schemas/), product config (config/), and docs (requirements/implementation) all live together.
    - Single package.json at the root lists every dependency (Expo, app libraries, and dev tools). No nested app with its own package.json.

🧭 App Shell and Navigation
    - Bottom tabs: Today, Build, Break, Track, Profile
    - Build and Break: separate list screens filtered by habit type
    - Visual language: build-oriented habits use a green theme; break-oriented use pink/red; informational accents may use blue (see design mock)
    - Track and subscription paywall: dark-mode presentation (per design mock)
    - Habit detail: tabs for Plan, Progress (stats for the habit from logs), and Plan History (version list; not the same as Track “habit history”)

👤 User Features
1. Authentication
    - Email/password signup & login
    - Google OAuth login
    - Persistent sessions
    - Logout (available from Profile)

2. Profile
    - View signed-in account (e.g. email) where available
    - Logout
    - Subscription status and entry to paywall / manage subscription
    - Restore purchases (RevenueCat)
    - Optional: legal links (privacy/terms) if required for store listing

3. Habit Creation
Users can create a habit with:
    - Name
    - Type: build or break
    - Start date
    - Optional context (used for AI generation)
    - There is no user-selected “habit difficulty” at creation. The first plan is generated at a default intensity; the user refines how easy or hard the plan feels only via plan adjustment (see §15)

4. End-to-end flow (MVP)
    - After creating a habit, the user is guided to generate a plan (no “skip plan” in MVP)
    - If generation fails, the habit may exist with no active plan until retry (current_plan_id may be null)

5. AI Plan Generation
Each habit has an AI-generated plan with:
    - Summary
    - difficulty (in plan JSON: easy, medium, or hard) — the model’s read on how demanding the plan is; not set by a separate user field at creation
    - Estimated duration (days)
    - Triggers (only for breaking habits; array of strings)
    - Steps (max 7)
    - Each step includes:
        - id: stable string (UUID) inside ai_plan JSON — required for HabitLog
        - title
        - description
        - frequency: daily or weekly
        - If weekly: weekdays — array of integers 0–6 where 0=Sunday, 1=Monday, …, 6=Saturday (align with JavaScript Date.getDay())

6. “Due today” and Today view
    - A step is “due today” on a given calendar date if:
        - frequency is daily: every day on or after the habit start date (in the user’s local day), or as constrained by product rules, and
        - frequency is weekly: today’s weekday is included in the step’s weekdays array (and the habit has started)
    - Today view lists only steps that are due today for each habit, grouped by habit

7. Plan Management
    - Generate plan after habit creation (MVP: required path)
    - Regenerate plan with optional user input (counts against AI generation quota for free users)
    - Store multiple plan versions
    - Mark one plan as active per habit
    - Versioning
        - Plans are versioned (v1, v2, v3…)
        - New plan version receives new step ids in ai_plan
        - Users can:
            - View past versions (premium)
            - Revert to a previous version / set active (premium)
    - Free users see a remaining-generation indicator (global pool) on generate/regenerate surfaces (see monetization)

8. Manual Step Editing (MVP-lite)
Users can:
    - Rename step (updates title in stored ai_plan; id unchanged for current version)
    - Delete step (removes from ai_plan for current version; logs reference step_id historically)
    - Mark step complete (see Today view; syncs to HabitLog)
(No full plan editor—keep lightweight)

9. Today View (Primary Screen)
Displays:
    - Steps due today across all habits
    - Grouped by habit
    - Checklist interaction

Actions:
    - Toggle completion
    - Quick access to habit details

10. Build Habits Page
    - List of habits with type = build
    - Add new habit
    - View streaks
    - Navigate to details screen for each habit

11. Break Habits Page
    - Same as Build, filtered for type = break

12. Tracking System
Track:
    - Step completion per day (HabitLog)
    - Completion logs keyed by habit_id, step_id, and local date

13. Streak System (MVP Critical)
    - Definition: For a given habit, a streak is the count of consecutive local calendar days, ending on the most recent counted day, such that for each of those days every step that was due on that day was completed (per HabitLog for the corresponding step_id and date).
    - Display: per habit, plus an overall or highlighted streak on Today as appropriate
    - Steps not due on a day do not break the streak; partial completion on a day when any step was due does break the streak for that day

14. Tracking Page
Displays:
    - Habit- and app-level history
    - Completion rate, streaks
    - Weekly / monthly overview (per mock)

15. Plan Adjustment (AI Feedback Loop) — primary way to calibrate plan intensity
    - After a plan exists, the user says how the plan felt: “Too easy”, “Just right”, or “Too hard”
    - Optional free-text feedback (e.g. why it felt off, constraints)
    - The Edge Function / prompt uses that signal (and optional text) to regenerate a plan of appropriate intensity: easier, similar, or harder as appropriate
    - “Just right” with no extra text may result in a minor revision or a no-op policy (product choice); if regenerating, it counts as one AI use for the free tier when applicable
    - Every regeneration (initial plan after create, open-ended regenerate, or this adjustment) follows the same generation quota rules where applicable

💰 Monetization (RevenueCat)
    - Free Tier
        - Max 3 habits
        - Limited AI generations via a global pool (not per-habit): 3 per calendar month (resets each month; source of truth: config/product.json; display “generations left” in UI)
        - No access to plan version history (past versions) except current active
        - Basic tracking
    - Paid Tier
        - Unlimited habits
        - Unlimited AI regenerations
        - Access to plan version history and switching active version / revert
        - “Advanced” adjustments (same pipeline; premium is primarily limits + history)
    - Paywall Triggers
        - Exceed habit limit
        - Exceed global generation limit for the period
        - Attempt to view or restore non-current plan history

Enforcement: limits must be enforced server-side (Edge Function and/or RLS/secure RPC) so the client cannot bypass them.

🧱 Data Models

Habit {
  id: string (uuid)
  user_id: string (uuid, matches auth.users)
  name: string
  type: "build" | "break"
  start_date: date
  current_plan_id: string | null
  created_at: timestamp
}

HabitPlan {
  id: string
  habit_id: string
  version: number
  ai_plan: JSON (see schema: ai_plan JSON contract)
  is_active: boolean
  created_at: timestamp
}

HabitLog {
  id: string
  habit_id: string
  step_id: string
  log_date: date
  completed: boolean
  created_at: timestamp (optional)
}

Profiles (optional but recommended) {
  id: string (uuid, equals auth.users id)
  display_name: string | null
  updated_at: timestamp
}

ai_generation_usage (or equivalent) {
  user_id: string
  month_key: string (e.g. YYYY-MM) or period identifier
  generations_used: number
  — unique (user_id, month_key) for monthly windows
}

🤖 AI Requirements
    - Must return strict JSON
    - Must follow the shared Zod schema in schemas/aiPlan.zod.ts (keep in sync with the Edge Function)
    - First-time plan generation: no user difficulty in the request—model chooses a reasonable default and sets `difficulty` in the JSON; regenerations and adjustments must include the user’s “too easy / just right / too hard” choice (and optional text) in the prompt
    - Must be actionable and realistic
    - Max 7 steps
    - Include triggers for breaking habits
    - Weekly steps must include weekdays 0–6

⚠️ Constraints
    - Plans are static (no auto-adaptive behavior)
    - AI runs only on generation/regeneration
    - No social features
    - No push notifications in MVP
    - No offline-first or sync conflict resolution in MVP (network required for mutations)
